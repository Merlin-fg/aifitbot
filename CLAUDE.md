# AIFitBot — 产品文档与开发规范

## 1. 产品概述

**产品名称：** AIFitBot  
**产品定位：** AI 私人健身教练，面向健身新手的专业知识问答平台  
**目标用户：** 健身初学者、需要饮食/训练指导的普通用户  
**核心价值：** 基于专业健身知识库的 RAG 智能问答，回答有据可查，弥补通用 AI 回答宽泛、不可靠的缺陷

## 2. 运行环境

- 支持 **Windows 10/11** 和 **macOS** 桌面运行
- 用户通过浏览器访问本地服务（`http://localhost:xxxx`）
- 无需 Docker，通过 Python 虚拟环境直接运行
- LLM 和 Embedding 均使用云端 API（阿里云百炼平台），不依赖本地 GPU

## 3. 功能需求

### 3.1 用户认证模块
| 编号 | 功能 | 描述 |
|------|------|------|
| U-01 | 用户注册 | 新用户填写用户名、密码完成注册 |
| U-02 | 用户登录 | 已注册用户通过用户名密码登录 |
| U-03 | 修改密码 | 登录用户可修改自己的密码 |
| U-04 | 角色区分 | 系统内置两种角色：管理员(admin) 和 普通用户 |
| U-05 | 管理员账号 | 默认管理员用户名 `admin`，密码 `123456`（首次登录后强制修改） |

### 3.2 知识库管理模块（仅管理员）
| 编号 | 功能 | 描述 |
|------|------|------|
| K-01 | 文档上传 | 支持上传 Markdown (.md)、PDF、TXT 文件 |
| K-02 | 文档列表 | 展示已上传的所有文档，支持预览内容 |
| K-03 | 文档删除 | 支持删除已上传的文档，同步删除向量数据 |
| K-04 | 向量化处理 | 上传后自动切分文档并生成向量，存入向量数据库 |
| K-05 | 处理状态提示 | 上传大文档时显示处理进度或状态 |

### 3.3 知识库问答模块（所有登录用户）
| 编号 | 功能 | 描述 |
|------|------|------|
| Q-01 | 知识库问答 | 用户输入问题，系统检索知识库后生成回答 |
| Q-02 | 引用溯源 | 回答中标注引用了哪些知识库片段，用户可展开查看原文 |
| Q-03 | 流式输出 | AI 回答以打字机效果逐字显示，提升体验 |
| Q-04 | 对话记忆 | 多轮对话中保持上下文连贯 |

### 3.4 会话管理模块
| 编号 | 功能 | 描述 |
|------|------|------|
| S-01 | 多会话 | 每个用户可创建多个独立会话（如"增肌计划"、"减脂咨询"） |
| S-02 | 会话列表 | 侧边栏展示用户的所有会话，支持切换 |
| S-03 | 会话重命名/删除 | 用户可管理自己的会话 |
| S-04 | 历史持久化 | 所有对话记录存储到数据库，重新登录后可找回 |
| S-05 | 会话导出 | 用户可将对话记录导出为 Markdown 文件 |

### 3.5 性能与体验优化
| 编号 | 功能 | 描述 |
|------|------|------|
| P-01 | 向量库缓存 | 应用启动时加载，避免每次请求重建 |
| P-02 | LLM 连接池 | 复用 API 连接，减少握手开销 |
| P-03 | 响应流式输出 | SSE (Server-Sent Events) 实现逐 token 推送 |
| P-04 | 数据库索引 | 对高频查询字段建立索引 |
| P-05 | 前端资源压缩 | 静态资源 gzip 压缩传输 |

## 4. 技术选型（已确认 ✅）

| 模块 | 选择 | 说明 |
|------|------|------|
| LLM 框架 | **LangChain + LCEL** | 项目负责人指定 |
| LLM 提供商 | **阿里云百炼** (qwen-plus) | OpenAI 兼容接口 |
| 嵌入模型 | **阿里云百炼 text-embedding-v2** | 云端 API，不占本地资源 |
| 后端框架 | **FastAPI** | 原生异步，SSE 流式输出 |
| 前端方案 | **Jinja2 + HTMX + Alpine.js** | 纯 Python 全栈，无需 Node.js |
| 数据库 + ORM | **SQLite + SQLModel** | 零配置，Pydantic 类型安全 |
| 向量数据库 | **ChromaDB** (embedded) | 本地持久化，LangChain 原生集成 |
| 认证方案 | **JWT** (python-jose + bcrypt) | 无状态 Token 认证 |
| 包管理工具 | **uv** | 极速，已有 uv.lock |

> 以上选型于 2026-07-19 由项目负责人确认。

## 6. 架构分层原则

```
routes (表现层)  →  services (业务层)  →  repositories (数据层)  →  models (数据模型)
```

- **routes**: 处理 HTTP 请求/响应，参数校验，权限检查。不包含业务逻辑。
- **services**: 业务逻辑、编排调用。不直接操作数据库。
- **repositories**: 封装数据库和向量库的 CRUD 操作。
- **models**: Pydantic/SQLModel 数据模型定义。

依赖规则：上层可以调用下层，下层绝不能调用上层。

## 7. 开发协作规范

### 7.1 技术决策流程（核心规则）

> **在整个项目开发过程中，所有技术相关决策必须遵循以下流程：**
>
> 1. 开发者(Claude) 列出 2-3 个可选方案，每个方案附带优劣势说明和推荐理由
> 2. 项目负责人(Merlin-fg) 审阅方案并做出选择
> 3. 开发者按照选择执行，不可自行决定技术方案
>
> **此规则适用于：** 框架选择、库选择、架构设计、数据库设计、API 设计、部署方案、以及任何影响项目走向的技术决策。
>
> **不适用：** Bug 修复、代码格式调整、变量命名等不影响架构的细粒度实现细节。

### 7.2 代码规范

- Python 3.13+，使用 type hints
- 所有函数包含 docstring（Google 风格）
- 使用 `logging` 模块记录关键操作
- 敏感信息（API Key、密码）不硬编码，使用 `.env`
- 每完成一个模块，必须可独立验证（运行并通过基本测试）

### 7.3 提交规范

- 一个功能模块一个 commit
- Commit message 格式：`类型: 简短描述`（如 `feat: 添加用户注册接口`）
- 类型：`feat` / `fix` / `refactor` / `docs` / `test` / `chore`
